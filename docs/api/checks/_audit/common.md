# Domain: common

## Summary
9 endpoints audited (GET /home, /galgame, /comment, /resource, /resource/:id, /ranking/user, /ranking/patch, /hikari, /moyu/patch/has-patch). All GETs tested live and shapes diffed against the FE `.d.ts` + page/component consumers. Shapes all match the frontend contract (paginated `{items,total}` for /comment & /resource; `{galgames,total}` for /galgame; bare `GalgameCard[]` for /ranking/patch; `RankingUser[]` for /ranking/user; `HomeResponse` for /home). One real **HIGH** security issue: the public bulk list feeds (`/resource`, `/home`, and the recommendations slice of `/resource/:id`) emit the full download payload (`content` download URLs + `code` + `password` + `s3_key`) for every row, with no auth and no rate-limit — defeating the dedicated rate-limited `/patch/resource/:id/link` reveal endpoint (router.go:82-86) whose entire stated purpose is to break bulk URL scraping. The FE list card (`app/components/resource/Card.vue`) does not consume any of these fields, so it is pure over-send. One **low** note on `/hikari` (external API) only stripping `content` for `s3` storage, leaving netdisk `content`/`code`/`password` exposed. No false positives worth reporting were dropped (verified the `like`→`like_count` remap is dead-but-harmless, and the ranking banned-user `continue` is correctly scoped).

## Endpoints

### GET /home — CommonHandler.GetHome
- verdict: fix
- tested: `curl /home` → `{code:0,...,data:{galgames:[7],resources:[2],comments:[5]}}`. galgames=7 (<12) and resources=2 (<6) due to the documented sfw-default NSFW post-filter. Top-level keys `galgames/resources/comments` match `HomeResponse` (home.d.ts:1). GalgameCard fields match patch.d.ts; resources match resource.d.ts (`note_html`, `patch` summary present); comments match comment.d.ts (`content_html`, `patch`).
- issues:
  - [high][security] The `resources` slice ships the full download payload for each row: `content` (real download URLs incl. `https://oss.moyu.moe/...` presigned S3 + `https://pan.baidu.com/...?pwd=` netdisk links) and `password`/`code`/`s3_key`. Public, unauthenticated, no rate-limit. EVIDENCE: handler.go:108,119,123,130 — home feed `Find(&resources)` then returns them via `RenderResourceNotes`/`response.OK` with no field stripping; resource.go model.go:169-174 marshals `code`/`password`/`content`/`s3_key`. Live: `/home` resource id=9023 `content="https://oss.moyu.moe/patch/6719/f8302..."`. FE `app/components/resource/Card.vue` never reads `content/code/password/s3_key`. FIX (BE): in `GetHome`, after building `resources`, blank `Content/Code/Password/S3Key` on every row before `response.OK` (a helper `stripResourceSecrets([]PatchResource)` reused across the list feeds). Download links must only come from the rate-limited `/patch/resource/:id/link`.

## Cross-cutting
- [high][security] Bulk download-payload over-send is a cross-endpoint pattern in this domain: `GetHome` (resources), `GetGlobalResources` (items), and `GetResourceDetail` (recommendations) all return `content`/`code`/`password`/`s3_key` that no FE consumer reads, defeating the rate-limited `/patch/resource/:id/link` design. FIX: add one shared helper in `internal/common/handler.go`, e.g. `func stripResourceSecrets(rs []patchModel.PatchResource){ for i:=range rs { rs[i].Content=""; rs[i].Code=""; rs[i].Password=""; rs[i].S3Key="" } }`, and call it on the home `resources`, global `resources`, and detail `recs` slices before responding. Leave the single `resource` in `/resource/:id` and the dedicated `/patch/resource/:id/link` untouched (they are the intended reveal surfaces). NOTE: `/patch/:id/resource` (patch domain) shares the same gap — flag for that audit.
