# Domain: auth

## Summary
5 endpoints audited: POST /auth/oauth/callback, POST /auth/logout, GET /auth/me, PATCH /auth/me, POST /auth/me/avatar. Two are OAuth display-layer proxies (PATCH /auth/me, POST /auth/me/avatar) that forward the user's Bearer and re-emit OAuth's envelope verbatim; the other three are native. All 5 verified correct against the FE contract (UserState in stores/userStore.ts, MeFull in pages/settings/user.vue). GET /auth/me JSON matches MeResponse field-for-field (live-tested with the admin cookie). No real defects found. Three candidate concerns were investigated and dropped as false positives: (1) UserState has `muted_message_types` not in MeResponse — intentional, it is preserved client-side by setUser and never sourced from /auth/me; (2) PATCH /auth/me returns OAuth's richer payload (uuid/email/status/moemoepoint=8135) which differs from moyu's local shape — FE only reads `res.code` then re-fetches GET /auth/me, so the OAuth body is never consumed; (3) callback vs me derive roles from different sources (userInfo.Roles vs JWT roles claim) — both produce identical `["admin"]` and feed the same MeResponse.Roles field.

## Endpoints

### POST /api/v1/auth/oauth/callback — AuthHandler.OAuthCallback
- verdict: ok
- tested: live. Empty body → `{"code":40000,"message":"Code is required; CodeVerifier is required","data":null}` (validation, mapped to ErrBadRequest 40000). Fake code → `{"code":40000,"message":"OAuth authentication failed","data":null}` (ExchangeCode fails → ErrBadRequest). Banned path (10014 → ErrAccountBanned) untestable without a real banned OAuth account; statically verified: handler/handler.go:60-65 maps service.ErrUserBanned → errors.ErrAccountBanned, and FE pages/auth/callback.vue:41 redirects to /account-banned on code 10014. Success path success-tested indirectly via GET /auth/me (same composeMe output). A real success run needs a live OAuth code (untestable safely).
- notes: success response is composeMe(local, userInfo.Sub, userInfo.Roles); FE callback.vue:53 does `userStore.setUser(res.data)` then navigates to `/user/${res.data.id}/resource`. `res.data.id` present in MeResponse (json:"id"). Validation request shape `{code, code_verifier}` (dto.go) matches FE body keys (callback.vue:32-33). last_login_time + ip stamped in a best-effort goroutine (handler.go:75-80) — non-blocking, correct columns per model.go.

### POST /api/v1/auth/logout — AuthHandler.Logout
- verdict: ok
- tested: NOT executed against the admin cookie — logout would DESTROY the audit session (Redis key delete + cookie clear), invalidating the credential used for the rest of this audit. Ready curl for human below. Statically verified: reads `moyu_session` cookie, looks up Redis session, fires `RevokeOAuthToken(refresh_token)` in a goroutine (handler.go:118; correct token — docs note /oauth/revoke matches by refresh_token), then DestroySession deletes the Redis key + expires the cookie (auth.go:342-359). No auth middleware (public, by design — doc row moyu.post.md:28). Returns `{code:0,message:"Logged out"}`. FE (UserDropdown.vue:30, MobileMenu.vue:88, settings/user.vue:166) only awaits it and then calls userStore.logout() locally — response body unused, so envelope shape is non-load-bearing.
- ready curl (human): `curl -i -X POST 'http://127.0.0.1:5214/api/v1/auth/logout' -H 'Cookie: moyu_session=<DISPOSABLE_SESSION>'` (use a throwaway session, NOT the admin audit cookie).

### GET /api/v1/auth/me — AuthHandler.Me
- verdict: ok
- tested: live with admin cookie → `{"code":0,"message":"OK","data":{"id":2,"sub":"97cadeaf-...","roles":["admin"],"name":"鲲","avatar":"https://image.kungal.com/avatar/user_2/avatar.webp","avatar_image_hash":"","bio":"...","moemoepoint":5,"daily_check_in":0,"daily_image_count":0,"daily_upload_size":0,"follower_count":0,"following_count":0}}`. No cookie → `{"code":40100,"message":"Please log in first","data":null}` (Auth middleware, correct). Every json tag in dto.MeResponse is present and matches UserState (userStore.ts:15-27) and MeFull (settings/user.vue:41-49) snake_case keys exactly. `moemoepoint` is moyu's LOCAL value (5), composeMe reads `local.Moemoepoint` (handler.go composeMe) — correct (OAuth's 8135 is a different ledger, not surfaced by /me). On OAuth /users/batch failure, composeMe returns local fields with empty name/avatar/bio (handler.go) — degrades gracefully, won't crash the page.

### PATCH /api/v1/auth/me — AuthHandler.UpdateMe
- verdict: ok
- tested: live. No cookie → `{"code":40100,...}` (auth middleware). Admin cookie + empty body `{}` (no-op, non-destructive) → HTTP 200 `{"code":0,"message":"成功","data":{"uuid":...,"name":"鲲","email":"kun@kungal.com","avatar":...,"bio":...,"moemoepoint":8135,"status":0,"is_anonymized":false,"roles":["admin"],"created_at":...}}` — confirms Bearer forwarded (OAuth resolved actor=user 2), status preserved, OAuth envelope re-emitted verbatim. Proxy correctness verified: proxyUserOAuth pulls access_token (401 if absent), forwards raw `c.Body()` + Content-Type to OAuth PATCH /auth/me, re-emits `c.Status(status).Send(raw)` (handler.go proxyUserOAuth). FE settings/user.vue:85-89 sends only changed `{name?, bio?}` (pointer-semantics partial update); BE comment says it accepts `{name, avatar, avatar_image_hash, bio}` — FE subset is valid. FE reads only `res.code` then refreshMe() (GET /auth/me), so the divergent OAuth body shape is never bound to UserState — no mismatch.
- ready curl (human, mutating name — REVERSIBLE if you restore the old name): `curl -X PATCH 'http://127.0.0.1:5214/api/v1/auth/me' -H 'Cookie: moyu_session=<SESSION>' -H 'Content-Type: application/json' -d '{"bio":"test"}'` (changes real OAuth bio; revert by re-sending the original bio).

### POST /api/v1/auth/me/avatar — AuthHandler.UploadAvatar
- verdict: ok
- tested: live. No cookie → `{"code":40100,...}`. Admin cookie + no file → HTTP 400 `{"code":8,"message":"缺少必要参数"}` — confirms multipart proxy reaches OAuth, Bearer forwarded, OAuth's 400 + original code 8 round-tripped with status preserved (proxyUserOAuth re-emits c.Status(status)). Multipart boundary survival verified statically: handler reads `c.Body()` (raw bytes, no prior c.FormFile call to consume the stream) and forwards `c.Request().Header.ContentType()` (includes boundary). FE settings/user.vue:111-122 builds FormData with field `file` and posts via raw `$fetch.raw` with credentials:include, inspecting `_data` envelope. Field-name match to OAuth (`file`) is OAuth-internal and not safely testable without a real file upload — marked by-design (proxy forwards verbatim; FE/OAuth field contract is consistent in the codebase).
- ready curl (human, REAL upload — only run with a disposable image): `curl -X POST 'http://127.0.0.1:5214/api/v1/auth/me/avatar' -H 'Cookie: moyu_session=<SESSION>' -F 'file=@/path/to/test.webp'` (mutates the real avatar via OAuth+image_service; not self-reversing).

## Cross-cutting
- none. Auth middleware correctly applied: oauth/callback + logout are public (by design — logout reads cookie directly, callback establishes the session), me/PATCH-me/avatar all use `auth` (login required). No IDOR surface — every authed endpoint acts only on the session's own user (handler.go uses middleware.MustGetUser(c).ID / GetAccessToken(c), never a body/path-supplied user id). No moemoepoint/notification/counter side-effects fire in this domain (none expected). Proxy endpoints correctly forward the end-user JWT (not Client Basic Auth) per docs/oauth/02-user-profile.md 代理模式.
